## Bug found & fixed (the one real defect)

**Skip link did not move keyboard focus.** WCAG 2.4.1 (Bypass Blocks). The "Skip to
content" links point to `#main`, but the `#main` containers had **no `tabindex`**. A
fragment link to a non-focusable element scrolls the viewport but, in **Safari** (a
primary browser for this Apple-ecosystem product) and several screen readers, does
**not** relocate keyboard focus — so the next Tab press jumps straight back into the
nav, defeating the entire purpose of the skip link. Since skip-to-content / full
keyboard navigation *is* fix #6 of this task, this was a hole squarely in scope.

Fix (`5e8f509`):
- Added `tabindex="-1"` to every `#main` target across all 5 pages so activation
  reliably moves focus into the main region.
- Added `#main:focus{outline:none}` — the pre-existing `[tabindex]:focus-visible` rule
  would otherwise draw a 2px teal outline around the **entire** main region when the
  skip target receives programmatic focus. The outline is correct for real interactive
  controls but visually wrong for a full-width landmark; suppressed only for `#main`.

Verified after fix: all `#main` elements carry `tabindex="-1"`, CSS rule present, and
all pages/assets still serve 200.
